### 第 2 章 — 測試案例

每個測試案例建議包含以下欄位：

- **案例編號**：TC-001, TC-002, ...（格式固定，方便 AI 識別）
- **案例名稱**：簡短描述測試目的（例：「使用者登入成功」）
- **測試步驟**：編號清單，每步驟一行
- **預期結果**：測試通過的判斷依據

## 三、範例格式 B：條列式

### TC-001 使用者登入成功

**測試步驟：**
1. 前往 /login 頁面
2. 輸入電子郵件 test@example.com 到信箱欄位
3. 輸入密碼 password123 到密碼欄位
4. 點擊「登入」按鈕
5. 確認轉跳到 /dashboard

**預期結果：** 使用者成功登入，頁面顯示「歡迎回來」文字

---

### TC-002 空白表單提交驗證

**測試步驟：**
1. 前往 /login 頁面
2. 直接點擊「登入」按鈕（不填任何資料）
3. 確認頁面出現驗證錯誤提示

**預期結果：** 表單顯示「電子郵件不可空白」、「密碼不可空白」提示訊息

## 五、常見測試類型範例

### 表單驗證測試
```
TC-010 必填欄位驗證
步驟：
1. 前往 /register
2. 直接點擊「送出」按鈕
3. 確認可見 text=此欄位為必填

預期結果：所有必填欄位顯示紅色錯誤提示
```

### 導覽測試
```
TC-020 主選單連結測試
步驟：
1. 前往 /
2. 點擊 nav .menu-item:first-child
3. 確認網址 包含 /about

預期結果：成功導覽至「關於我們」頁面
```

### 搜尋功能測試
```
TC-030 關鍵字搜尋
步驟：
1. 前往 /search
2. 輸入 測試商品 到 #search-input
3. 點擊 #search-btn
4. 確認可見 text=搜尋結果

預期結果：頁面顯示相關搜尋結果
```

## 六、注意事項

1. **不需要寫程式碼**：只要用中文描述操作步驟即可
2. **CSS 選擇器**：若您知道按鈕的 ID 或 class，可以直接寫（如 `#login-btn`），否則用文字描述（如「登入按鈕」）
3. **URL 路徑**：步驟中的網址可以只寫相對路徑（如 `/login`），系統會自動加上 Base URL
4. **截圖會自動產生**：不需要特別加截圖步驟，系統會在每個操作後自動截圖
5. **測試資料**：測試用的帳號密碼、填寫數值請直接寫在步驟中（測試環境用）
